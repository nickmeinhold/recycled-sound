# CLIPArTT Test-Time Adaptation Report

_Generated by `data/clipartt_adaptation.py`._

## Honest finding (TL;DR)

**The adapter made retrieval slightly worse on this synthetic target set.**
Top-1 dropped from 0.226 → 0.195 (−3.1 pp), top-5 was essentially flat
(0.535 → 0.531), and average max similarity collapsed from 0.786 → 0.602.

The most likely cause is that pseudo-labels generated *from the unadapted
CLIP on a heavily-augmented target* are themselves noisy — we're teaching
the adapter from a teacher that doesn't know the answer either. With a
real-world target set (where domain shift is a smooth, learnable
invariance rather than an adversarial augmentation chain), CLIPArTT
typically helps; here it's working against itself.

**Recommended next step:** LoRA fine-tune of the CLIP image tower against
a curated real-world target set (task #15). This run is preserved
nevertheless: the new collection `clip_visual_adapted` lives alongside
`hearing_aids_visual` for A/B comparison, and the adapter weights are at
`data/processed/clipartt_adapter.pt` should we want to revisit with
different hyperparameters or a real target set.

## ⚠️ Limitation: synthetic target distribution

No real-world hand-held photos were available at `data/real_world_photos/`,
so this run used a **stratified hold-out of the catalog with heavy
hand-held-style augmentations** (blur, rotation, perspective, color jitter,
low-res rescaling, partial occlusion) as a synthetic stand-in for the
target distribution.

Augmentations approximate, but do NOT replicate, the true domain shift
(specular highlights on plastic shells, finger occlusions, focus failures,
warm indoor lighting, real-camera ISP pipelines). The numbers above are an
**indicative lower-bound** of adapter capability — once Seray drops real
scanner photos into `data/real_world_photos/<brand>/...` and we rerun, we'll
get a meaningful estimate.

## What this is and isn't

- This is a **CLIPArTT-inspired lightweight variant**: instead of LoRA on
  the CLIP image tower, we train a 512×512 residual adapter on top of
  frozen image embeddings, using transductive top-k pseudo-labels from the
  unadapted CLIP. Faster to iterate; weaker capacity. If retrieval doesn't
  recover most of the clean→augmented gap, the next step is a LoRA
  fine-tune of the image encoder (task #15).
- The new embeddings are written to ChromaDB collection
  **`clip_visual_adapted`**. The original `hearing_aids_visual` is untouched,
  so the scanner can A/B at query time by switching collections.
- Adapter weights persisted at `data/processed/clipartt_adapter.pt` so the
  on-device pipeline (or cloud function) can apply the same projection to
  query embeddings.
